# v0.1 Checklist

v0.1 target: fresh node boot -> pair -> connect link -> screen appears, with no terminal use for the user.

## Implemented In Repo

- Cargo workspace and locked monorepo layout scaffold.
- `sw-core` protocol/config/pairing/capability types; real UUID validation.
- Shared certificate store in `sw-core` (self-signed generation, SHA-256 fingerprints, self-healing writes) used by both peers.
- `sw-agent` VA-API render-device enumeration and H.264 decode detection.
- `sw-agent` persistent identity store; paired-host trust persisted before the node marks itself paired.
- Agent API over HTTPS: `GET /v1/{health,capabilities,pairing,screen}`, `POST /v1/pairing`, `POST /v1/screen`; mTLS required after pairing.
- Pairing PIN/QR never exposed over the network — they reach the node's physical screen only, via the atomic kiosk state file.
- Unbiased OS-random pairing PINs; QR payload from runtime state.
- Node mDNS advertisement with runtime port + TXT metadata; companion browser parses typed nodes incl. addresses.
- Companion host identity/certificate store (`config.json` + host cert pair, UUID-keyed node trust).
- Companion pairing client: fingerprint-pinned TLS, PIN entry UI, trust recorded per node UUID.
- Companion Apollo layer: detect/configure (managed config block), random dashboard credentials, one-shot stream-PIN arming, service control — Apollo dashboard never shown.
- Companion Screen toggle end-to-end: Apollo prep → mTLS `connect`/`disconnect` → node kiosk streams.
- Node kiosk: pairing screen (unicode QR + PIN), idle screen, streaming state; supervised streaming client with one-shot inner pairing and exponential restart backoff; dev escape hatch behind an env flag.
- Auto-connect: debounced mDNS presence watcher; paired always-on nodes brought up on appearance, torn down on sustained absence, with UI notifications.
- Node image automation: live-build tree + `build.sh` (binaries, systemd units, Moonlight repo, VA-API drivers, preseeded safe dual-boot installer).

## Still Required For v0.1 Acceptance

- Build the ISO on a Debian environment and boot it on real hardware.
- Test the full flow on the first physical host/node pair (see `docs/PROFILE-dev-machine.md`), including:
  - pairing from the companion against a real kiosk screen,
  - Apollo config keys verified against the installed Apollo version (managed block lives in `companion/src-tauri/src/apollo.rs`),
  - Moonlight CLI pair/stream flags verified on the node,
  - auto-connect across USB-Ethernet unplug/replug.
- Fix-ups from hardware findings, then tag v0.1.

## Non-Negotiable UX Rule

Neither side exposes upstream setup as the product:

- Host: users use the SecondWind companion, not Apollo's browser dashboard.
- Node: users use the SecondWind node image, not Debian/Moonlight/cage/driver commands.

Debug logs may mention upstream internals for support, but normal product flows must explain issues in SecondWind terms. (The kiosk screens have a unit test enforcing that no upstream names appear.)
