# Architecture

SecondWind is a Windows companion plus a Debian node image. The companion and node agent hide upstream tools behind a SecondWind UX.

## Components

- `sw-core`: shared protocol, config, pairing, certificate store, kiosk contract, and capability types. The only place protocol/config types live.
- `sw-agent`: node daemon. Owns node state, capability detection, pairing, the kiosk state file, and the node API.
- `sw-kiosk`: node kiosk. Renders the pairing/idle/streaming screens fullscreen (under `cage` + `foot`) and supervises the streaming client with restart backoff.
- `sw-launcher`: host-side launch helper scaffold for the Apps phase (v0.3).
- `companion`: Windows companion (Tauri). Owns the host identity, discovery, pairing, the Apollo control layer, the Screen toggle, auto-connect, and user-visible errors.
- `node-image`: Debian live-build tree producing the bootable node ISO (live node + preseeded dual-boot installer).

## Product Boundary

Apollo, Moonlight, Debian, cage, VA-API tooling, and system services are implementation details. Users see SecondWind screens, SecondWind pairing, and SecondWind errors. Debug logs may mention upstream internals for support; product flows must not.

## Trust model

- Both peers keep a self-signed certificate (shared store in `sw_core::certificates`); identity is the SHA-256 certificate fingerprint.
- Node → host trust: established during PIN pairing; the agent persists the host certificate *before* marking itself paired, then requires that client certificate for every TLS connection (mTLS).
- Host → node trust: the companion pins the node fingerprint. Before pairing the pin comes from the node's mDNS TXT / QR; after pairing it always comes from the persisted per-UUID trust, never from the network.
- The pairing PIN and QR payload are proximity proofs. They travel exactly one path: agent → kiosk state file → the node's physical screen. The network API never returns them.
- The one-shot *stream* pairing PIN (node's streaming client ↔ host's streaming server) is generated by the companion, armed on Apollo's localhost API, and forwarded to the node inside the mTLS screen-connect call.

## Agent ↔ kiosk contract

The agent owns node state; the kiosk owns the physical screen. They meet at a JSON state file (`sw_core::kiosk::KioskState`, path from systemd `RuntimeDirectory`, tmpfs-backed). The agent rewrites it atomically on every transition; the kiosk polls it. States: `starting`, `unpaired` (QR + PIN), `idle`, `streaming` (host address + optional one-shot stream PIN).

## v0.1 Flow

1. The node boots into minimal Debian; `sw-agent` detects capabilities, generates/loads identity + certificate, and advertises `_secondwind._tcp.local.` with its runtime port and fingerprint once the HTTPS listener is bound.
2. `sw-kiosk` shows the pairing screen (QR + PIN) from the kiosk state file.
3. The companion discovers the node, the user enters the PIN, and the companion POSTs the pairing request over fingerprint-pinned HTTPS with its host certificate in the body.
4. The agent verifies the PIN, persists host trust, flips to mTLS, and the kiosk switches to the idle screen.
5. Screen on (toggle or auto-connect): the companion configures Apollo (managed config block, random dashboard credentials, service running), arms a one-shot stream PIN if this node never stream-paired, and sends `connect` to the agent over mTLS.
6. The agent records the requesting host's address and publishes `streaming` to the kiosk; the kiosk runs the streaming client (pair first if a PIN was supplied), supervised with exponential backoff.
7. Auto-connect: a companion watcher browses mDNS every few seconds with debounced presence (a single missed scan — e.g. a USB Ethernet adapter blink — does not bounce the session). Paired + always-on nodes are brought up on appearance; sustained absence tears down in reverse feature order and notifies the user.
8. Screen off: `disconnect` over mTLS; the kiosk stops the client and returns to idle; Apollo destroys the virtual display when the stream ends, so Windows reflows like a monitor was unplugged.

## API

JSON over HTTPS, versioned under `/v1/`, mutual TLS after pairing. The agent binds the address/port given by configuration (ephemeral port by default in the image) — never a source constant.

| Route | Method | Purpose |
|---|---|---|
| `/v1/health` | GET | service status (`Ready`/`Degraded` from H.264 decode presence) |
| `/v1/capabilities` | GET | node UUID + detected capabilities |
| `/v1/pairing` | GET | pairing status (never the PIN/QR) |
| `/v1/pairing` | POST | pairing request: host name + host certificate + PIN |
| `/v1/screen` | GET | screen state (`not_paired` / `idle` / `streaming`) |
| `/v1/screen` | POST | `connect` (with optional one-shot stream PIN) / `disconnect`; the agent uses the requesting peer's address as the stream target |
| `/v1/disk` | GET | disk state (`not_paired` / `unavailable` / `ready` / `exposed` + target) |
| `/v1/disk` | POST | `enable` / `disable` the export of the designated data partition |

## Disk (v0.2)

- At install, the SecondWind partition recipe labels the node's data partition `SECONDWIND_DATA`. On first boot a provisioning unit generates `/etc/secondwind/disk.env`: device path, per-node IQN, and random CHAP credentials.
- The agent starts/stops the `secondwind-disk` systemd unit (a polkit rule scopes the unprivileged agent to exactly that unit); the unit's script builds the LIO iSCSI target for the designated partition only, CHAP-authenticated.
- The CHAP credentials travel exclusively inside the mTLS `exposed` response to the paired host — the block layer is protected by a secret only the paired host can learn.
- The companion drives the Windows initiator through bundled PowerShell scripts: attach (with first-use GPT/NTFS initialization of the SecondWind disk only), drive-letter assignment from per-node config, and flush + detach.
- Ordering: bring-up is screen → disk; teardown is disk (flush first) → screen. On link loss, the local initiator session is still flushed and cleaned using the last-known IQN.

## Apps (v0.3)

- **Policy engine** (`sw-launcher`): per-app *Always on node / Always local / Ask each time* + *fallback-to-local*, resolved against live node availability (reachable / off-but-wakeable / unreachable). Pure and unit-tested; "ask" bubbles to the UI.
- **Node session**: the image runs one xpra session (`sw-xpra.service`) with a per-boot random password. The agent's `GET /v1/apps` returns the session endpoint (mTLS only) plus which whitelisted apps are installed; `POST /v1/apps` launches by `app_id` against the node's own catalog (`/etc/secondwind/apps.json`) — host-supplied command lines are never executed.
- **Seamless windows**: the companion attaches the host xpra client to the session (`sw_launcher::attach_spec`), so node apps appear as native host windows.
- **Files**: the host exposes one dedicated folder over SMB (`Enable-SecondWindShare.ps1`: dedicated `SecondWindShare` local account, random password, elevation once). Before any node launch the companion sends the UNC + credentials over mTLS; the agent's polkit-scoped `secondwind-share` unit mounts it via CIFS. Apps read/write user files through the mount; nothing persists on the node.
- **Cache-and-sync**: catalog entries with `synced_profile` run through `secondwind-run-synced.sh` — profile copied from the share to node tmpfs at session start (single-instance flock), `$HOME` redirected, atomically staged + swapped back on exit.
- **Wake-on-LAN**: the agent detects interface MACs (`/sys/class/net`); the companion stores them at pairing. A launch against a powered-off node sends magic packets, waits for mDNS reappearance (pinning stored trust, not discovery), then proceeds.

| Route | Method | Purpose |
|---|---|---|
| `/v1/apps` | GET | session endpoint + installed apps (paired only) |
| `/v1/apps` | POST | launch by whitelisted `app_id` |
| `/v1/share` | GET | share mount state |
| `/v1/share` | POST | configure + mount the host share |

## Capability Detection

Capability detection inspects the running machine and never assumes GPU model, render device order, codec support, panel modes, interface names, paths, or addresses. The VA-API probe enumerates `/dev/dri/renderD*`, runs `vainfo` per node, and counts H.264 decode only on `VAEntrypointVLD`. (Phase 0 proof: the working decoder was not the first render device.)

## Discovery

The node advertises `_secondwind._tcp.local.` only after the listener is bound, so the advertised port is runtime state. TXT records carry the node UUID, display name, certificate fingerprint, and API version. The companion browses with the same shared constants and parses into typed nodes, including resolved addresses (IPv4 preferred) for connection attempts.

## Host side (companion)

- Host state under the app-data dir (or `SECONDWIND_COMPANION_STATE_DIR`): `config.json` (`sw_core::SecondWindConfig`, UUID-keyed nodes with trust + screen settings) plus the host certificate pair.
- Apollo layer: detection (env override → Program Files candidates), a marker-delimited managed config block (merge preserves foreign lines), random dashboard credentials set via CLI and stored locally, PIN arming over the localhost API, service start via `sc`. Screen defaults to always-on at pairing so auto-connect works with zero clicks.
